# How to Contribute

We welcome contributions to our documentation! This guide explains how you can help improve our docs for all products and features.

## Quick Start

The easiest way to contribute is directly from the documentation page:

1. Click the **"Edit this page on GitHub"** link at the bottom of any doc page
2. GitHub will automatically fork the repository for you
3. Make your changes in the web editor
4. Submit a Pull Request


## What Can You Contribute?

- **Fix typos and grammar** - Every improvement helps
- **Clarify confusing sections** - If something is unclear to you, it's probably unclear to others too
- **Add examples** - Code samples, use cases, or real-world scenarios
- **Translate content** - Help make docs available in more languages
- **Add new topics** - Missing documentation? Add it!
- **Document upcoming features** - Help document products that are "coming soon"

## Contribution Workflow

### Step 1: Fork the Repository

If you haven't already, fork the [docs repository](https://github.com/flomks/weareo.fun) on GitHub.

### Step 2: Create a Branch

Create a new branch for your changes:

```bash
git checkout -b fix-typo-in-authentication
```

### Step 3: Make Your Changes

Edit the MDX files in the appropriate product folder (e.g., `content/docs/ollm/` for OLLM docs). You can use:

- **Markdown** for text formatting
- **MDX Components** like `<Callout>`, `<CodeBlock>`, and `<ApiExample>`
- **Frontmatter** for metadata (title, description)

```mdx
---
title: "Your Page Title"
description: "A brief description"
---

# Your Content

<Callout type="warning">
  Important information here!
</Callout>
```

### Step 4: Commit and Push

```bash
git add .
git commit -m "Fix typo in authentication guide"
git push origin fix-typo-in-authentication
```

### Step 5: Open a Pull Request

1. Go to your fork on GitHub
2. Click "New Pull Request"
3. Describe your changes
4. Submit the PR

## Documentation Structure

Our documentation is organized by product:

```
content/docs/
  ├─ ollm/              # OLLM documentation
  ├─ [product-name]/    # Other product docs
  └─ how-to-contribute.mdx  # This file
```

When contributing to a specific product, place your files in the corresponding folder. For new products or "coming soon" features, create a new folder following the same structure.

## MDX Guidelines

### Available Components

Our documentation supports a wide range of MDX components to create rich, interactive content. Below is a comprehensive guide to all available components.

#### Callout

Highlight important information, warnings, or errors.

**Syntax:**
```text
<Callout type="info">
  This is an informational message.
</Callout>

<Callout type="warning">
  This is a warning message.
</Callout>

<Callout type="error">
  This is an error message.
</Callout>
```

**Props:**
- `type` (optional): `"info"` | `"warning"` | `"error"` (default: `"info"`)
- `children`: The content to display

#### CodeBlock

Display syntax-highlighted code blocks with copy functionality.

**Syntax:**
```text
<CodeBlock language="typescript">
const example = "Your code here";
</CodeBlock>
```

**Props:**
- `language` (optional): Programming language for syntax highlighting (e.g., `"typescript"`, `"javascript"`, `"bash"`, `"python"`, `"json"`, `"yaml"`) (default: `"text"`)
- `children`: The code to display (must be a string)

**Note:** The CodeBlock component includes a copy button that appears on hover.

#### Snippet

Display simple code snippets without syntax highlighting (useful for commands or short code examples).

**Syntax:**
```text
<Snippet text="npm install @ofoundation/ollm" dark />
```

**Props:**
- `text` (required): The code text to display
- `dark` (optional): Use dark background (default: `false`)
- `language` (optional): Language identifier (default: `"bash"`) - Note: This prop exists but Snippet doesn't use syntax highlighting

**Note:** The Snippet component also includes a copy button that appears on hover.

#### ApiExample

Display API endpoint examples with method badges.

**Syntax:**
```text
<ApiExample method="GET" endpoint="/api/v1/users" />
<ApiExample method="POST" endpoint="/api/v1/auth/login" />
```

**Props:**
- `method` (required): HTTP method (e.g., `"GET"`, `"POST"`, `"PUT"`, `"DELETE"`)
- `endpoint` (required): API endpoint path

#### Image

Display images with automatic handling for local and external URLs.

**Syntax:**
```text
<Image src="/docs/ollm/images/example.png" alt="Description" />
```

**Props:**
- `src` (required): Image path (local path starting with `/` or external URL starting with `http`)
- `alt` (optional): Alt text for accessibility
- `width` (optional): Image width (default: `1200`)
- `height` (optional): Image height (default: `600`)

**Note:** You can also use standard Markdown image syntax: `![Alt text](/path/to/image.png)`

For external images, use full URLs (e.g., `https://example.com/image.png`). For local images, place them in the `public/docs/` directory and reference them with paths starting with `/docs/`.

#### Tabs

Create tabbed content sections (useful for showing different installation methods, code examples, etc.).

**Syntax:**
```text
<Tabs items={['Tab 1', 'Tab 2', 'Tab 3']}>
  <Tab>
    Content for Tab 1
  </Tab>
  <Tab>
    Content for Tab 2
  </Tab>
  <Tab>
    Content for Tab 3
  </Tab>
</Tabs>
```

**Props:**
- `items` (required): Array of tab labels
- `defaultTab` (optional): Index of the default active tab (default: `0`)
- `children`: Tab components (must match the number of items)

**Note:** The Tabs component creates interactive tabbed sections. When rendered, users can click between tabs to switch content. The `items` prop must be a JavaScript array of strings.

#### Accordion

Create expandable/collapsible content sections.

**Syntax:**
```text
<Accordion title="Click to expand">
  Hidden content that appears when expanded.
</Accordion>
```

**Props:**
- `title` (required): The clickable title text
- `defaultOpen` (optional): Whether the accordion is open by default (default: `false`)
- `children`: The content to display when expanded

#### Button

Create styled buttons or links.

**Syntax:**
```text
<Button href="/docs/guide">Go to Guide</Button>
<Button variant="primary">Click Me</Button>
```

**Props:**
- `href` (optional): Link URL (if provided, renders as a link)
- `onClick` (optional): Click handler (if provided, renders as a button)
- `variant` (optional): `"default"` | `"primary"` | `"secondary"` (default: `"default"`)
- `children`: Button text

#### Table

Display data in a table format with overflow handling.

**Syntax:**
```text
<Table>
  <thead>
    <tr>
      <th>Header 1</th>
      <th>Header 2</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Data 1</td>
      <td>Data 2</td>
    </tr>
  </tbody>
</Table>
```

**Note:** Use standard HTML table elements (`<thead>`, `<tbody>`, `<tr>`, `<th>`, `<td>`) inside the `<Table>` component.

#### Video

Embed videos with responsive aspect ratio.

**Syntax:**
```text
<Video src="https://www.youtube.com/embed/VIDEO_ID" title="Video Title" />
```

**Props:**
- `src` (required): Video embed URL
- `title` (optional): Video title for accessibility
- `width` (optional): Video width (default: `560`)
- `height` (optional): Video height (default: `315`)

**Note:** The Video component creates a responsive iframe with a 16:9 aspect ratio. Make sure to use the embed URL format (e.g., `https://www.youtube.com/embed/VIDEO_ID` for YouTube).

#### Steps

Create step-by-step guides with numbered steps.

**Syntax:**
```text
<Steps>
  <Step>First step instructions</Step>
  <Step>Second step instructions</Step>
  <Step>Third step instructions</Step>
</Steps>
```

**Props:**
- `Steps`: Container component (no props)
- `Step`: Individual step component (no props, just children)

#### Badge

Display small tags or labels.

**Syntax:**
```text
<Badge variant="success">Available</Badge>
<Badge variant="warning">Beta</Badge>
```

**Props:**
- `variant` (optional): `"default"` | `"success"` | `"warning"` | `"error"` | `"info"` (default: `"default"`)
- `children`: Badge text

#### Card

Display content in a card container.

**Syntax:**
```text
<Card title="Card Title">
  Card content goes here.
</Card>
```

**Props:**
- `title` (optional): Card title
- `children`: Card content

#### Alert

Display alert messages with success support (similar to Callout but includes "success" type and uses slightly brighter text colors).

**Syntax:**
```text
<Alert type="success">Operation completed successfully!</Alert>
```

**Props:**
- `type` (optional): `"info"` | `"success"` | `"warning"` | `"error"` (default: `"info"`)
- `children`: Alert content

**When to use:**
- Use `Callout` for general informational messages, warnings, and errors
- Use `Alert` when you need to display success messages or want slightly brighter text colors

### Standard Markdown Elements

All standard Markdown elements are supported and styled:

#### Headings

Headings automatically have proper spacing (no automatic borders):

```text
# Heading 1
## Heading 2
### Heading 3
```

#### Lists

**Unordered Lists:**
```text
- Item 1
- Item 2
  - Nested item (indent with 2 spaces or tab)
- Item 3
```

**Ordered Lists:**
```text
1. First item
2. Second item
3. Third item
```

**Nested Lists:**
Use indentation (2 spaces or tab) to create nested lists:
```text
- Main item
  - Sub-item 1
  - Sub-item 2
- Another main item
```

#### Horizontal Rule

Add manual dividers between sections:

```text
---

Content after divider
```

#### Code Blocks

Standard Markdown code blocks (with syntax highlighting):

Use three backticks followed by the language name:

```typescript
const example = "Code here";
```

**Note:** All code blocks (both `<CodeBlock>` component and standard Markdown) include a copy button that appears on hover.

#### Links

Standard Markdown links:

```text
[Link Text](https://example.com)
```

#### Images

Standard Markdown images:

```text
![Alt Text](/path/to/image.png)
```

### Component Best Practices

1. **Use Callout for important information** - Don't overuse, but highlight key points (info, warning, error)
2. **Use Alert for success messages** - When you need to show success feedback or want slightly brighter text colors
3. **Use CodeBlock for code examples** - Provides syntax highlighting and copy functionality for multi-line code
4. **Use Snippet for commands** - Simple, one-line commands work best with Snippet (no syntax highlighting)
5. **Use Tabs for alternatives** - Great for showing different installation methods or code examples
6. **Use Accordion for optional details** - Keep main content visible, hide advanced details
7. **Use Steps for tutorials** - Perfect for step-by-step guides
8. **Use Badge for status indicators** - Quick visual indicators for availability, status, etc.

### Component Combinations

You can combine components for richer content. For example, you can nest Tabs inside Cards, or use Callouts within Accordions.

### Writing Style

- Be clear and concise
- Use examples when possible
- Link to related documentation
- Keep paragraphs short
- Use consistent terminology

### Frontmatter

Every MDX file should start with frontmatter:

```yaml
---
title: "Page Title"
description: "Brief description for search and previews"
---
```

## Contributing to "Coming Soon" Products

If you're contributing documentation for products that are still in development:

- Use the `<Callout type="warning">` component to indicate beta/early access status
- Clearly mark experimental features
- Include version information when relevant
- Link to related stable documentation

## Localization

If you're contributing translations:

- Create a file with the language suffix: `filename.de.mdx` or `filename.es.mdx`
- Keep the same structure and frontmatter
- Translate the content while maintaining technical accuracy
- Place translated files in the same folder as the original
